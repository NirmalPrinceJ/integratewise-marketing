1. Landing
When a user first lands on the enterprise application (e.g., the marketing or login page), the system primarily serves static content and basic scripts. At this stage, there are no back-end integration processes or data flows yet. Key points:

Logical completeness: The landing step is straightforward and needs no additional integration logic. The next meaningful step begins when the user clicks a CTA to proceed to the app’s login/signup.

2. Authentication (User Login via PKCE)
Upon clicking “Get Started” or “Log In” on the landing page, the user is directed to the application’s authentication mechanism. In this architecture, Supabase PKCE (Proof Key for Code Exchange) is used for user authentication[3]. Key technical actions:
•	PKCE OAuth Flow: The application uses a secure OAuth2/OIDC flow with PKCE for login and sign-up. The user is presented with a login form (or an SSO option) in the UI, and authenticated requests are routed through the BFF (Experience API). Supabase’s authentication service handles the details of this flow behind the scenes (as a managed Identity System API). The PKCE ensures that the exchange of the authorization code for a token is secure and not interceptable.
•	Token Issuance: After successful authentication (via credentials or third-party identity provider), Supabase issues a JWT access token for the user session[4]. This token contains the user’s identity and is stored on the client side (browser) and used for authenticated requests.
•	Onboarding Flag Check: The backend (via the BFF (Experience API)) inspects the user’s profile (e.g., user_metadata in Supabase) to determine if the user’s onboarding is complete[5]. This flag (often onboarding_complete) is set to false for new users and will be updated to true once they finish onboarding (see step 3). The application uses this flag to route the user either into the Onboarding flow (if not complete) or directly to the main Workspace dashboard if they have already onboarded[6].
•	Session Establishment: The JWT and user identity are used by the front-end’s API client to include an Authorization: Bearer <token> header on subsequent requests to the BFF (Experience API)[7] [8]. The user now has an active session.
Logical completeness:

Essentially, the platform has a library of canonical schemas (or “Spine” configurations) for combinations of industry and department – for example, a Sales domain in a Technology industry might include entities like Account, Contact, Opportunity, Lead, etc., with their standard fields.
o	After this screen, the system may perform “progressive hydration” of the workspace: the partial info collected so far can be used to start setting up the environment in the background[20]. For instance, right after the Profile step, the system could initiate workspace.initialize (as mentioned above) to save the config and perhaps begin preparing the data model, even before the user finishes all onboarding steps. This ensures that by the time the user completes onboarding, the backend work (like schema preparation) has already started.
•	Screen 2 – Goals & Workspace Details: Collects additional info such as the user’s goals for using the platform, naming for the workspace, the intended workspace type (e.g. demo vs production usage), and possibly an option to link the Knowledge UI (Flow C’s compounding knowledge base) to the structured data (Entity 360) for enhanced AI context[21] [22]. Technically:

o	Each connector in the catalog is tagged with metadata such as supported industries and departments (in MuleSoft terms, these are analogous to System API capabilities: each connector corresponds to a system, and the metadata describes what data domains it provides). o	At this point, no data is fetched from external systems yet; the user is simply selecting which systems to connect. The selection will determine which external API connections (System APIs) need to be authorized and which data pipelines (Process APIs) will run.

4. Connector OAuth Handshake & Token Storage
Once the user selects one or more connectors to integrate, the system proceeds with connecting to each external system using OAuth (for cloud APIs) or other appropriate authentication methods. This step involves BFF (Experience API) and System API interactions for each connector:
•	OAuth Authorization (Connector Integration): For each selected external application (e.g., Salesforce, HubSpot), the user initiates an OAuth 2.0 authorization sequence by clicking “Connect” in the UI. The front-end (via the BFF (Experience API)) redirects the user to the external provider’s authorization page (for OAuth Authorization Code grant). The user signs in to the external service and grants access to the enterprise application.
•	Callback and Token Exchange: After consent, the external service redirects back to a predefined callback endpoint in the enterprise application (e.g., /oauth/callback/:provider) with an authorization code. The application’s back-end (likely via a dedicated System API for that connector) uses the authorization code, along with a PKCE verifier or client secret as appropriate, to request an access token (and refresh token) from the external system[32]. This exchange is done securely server-to-server. For example, a Salesforce System API would hit Salesforce’s token endpoint with the code to get an OAuth token.
•	Secure Token Storage: The obtained OAuth tokens are encrypted and stored securely in the platform’s internal storage – for instance, in the connected_connectors section of the tenant_spine_config or a related secrets store. The integration will not expose these tokens to the client; instead, they are kept on the server for use by the integration services when making API calls to the external systems. This follows security best practices by treating tokens as sensitive credentials.
•	Connector Registration: The platform now marks the connector as “connected” for the tenant. In the tenant_spine_config (or a related table), the list of active connectors (connected_connectors) is updated to include the newly authorized systems[33]. This ensures that the integration knows which external sources are available for syncing data.
•	Trigger Initial Data Sync: A critical runtime action occurs here which must be explicitly noted: immediately after each connector’s OAuth is completed and tokens are stored, the system triggers an initial data synchronization for that connector[34]. In the given sequence, this action was implied but not detailed – it’s the step where the integration begins pulling data from the external system. In practice, the front-end or back-end invokes a function (e.g., triggerInitialSync(provider)) which calls into the Loader/connector Process API to start retrieving data from the external system using the new token[35]. This initial sync often targets a limited dataset (a “Phase 1 – Creamy” sync, designed to fetch the most critical records quickly, typically within 60 seconds)[36]. Without this trigger, the connected applications’ data would not actually start flowing into the system, so it is a necessary runtime step.
Logical completeness: The connector authorization step as described is correct. The main addition is to explicitly include the initial sync trigger after the OAuth handshake. This ensures that once a connector is authorized, the integration doesn’t wait idle – it proactively starts pulling data (using the new access token) in order to begin populating the workspace without delay[37].

5. Adaptive Schema Generation (Initialize Spine & Schema Router)
At this point, the user’s tenant has two key pieces of information: (a) the baseline Spine schema (from the Schema Picker in onboarding), and (b) the list of connected external systems. The platform now combines these inputs to fully initialize the tenant’s Adaptive Schema and ensure the data pipeline is configured to handle the incoming records:
•	Seeding the Spine Schema Registry: The BFF (Experience API) triggers the workspace.initializeSpine API (also referred to as initializeAdaptiveSpine) to finalize the tenant’s canonical schema[38]. This call passes the primary domain (department) selected, the industry, and the list of connected connectors. The back-end Process API handling initializeSpine will populate the Spine Schema Registry (spine_schema_registry) with the initial set of entity types and fields that define the tenant’s single source of truth (SSOT) data model[39] [40]. In effect, it creates “initial observations for primary domain and connectors” in the schema registry, meaning it registers the expected entities and key fields that the connectors will bring in, according to the baseline industry/department schema plus any known specifics of the chosen connectors[41].
o	This step is crucial for canonical schema hydration: it’s when the system’s canonical data model for the tenant is first instantiated and tailored to the tenant’s context. The integration platform either uses pre-defined schemas (for the given industry/domain) or may leverage connector metadata to augment the schema. “Connector metadata” refers to the knowledge of what objects and fields each external connector can provide. In a MuleSoft API-led approach, this could be achieved by calling the external system’s metadata API (e.g., describing Salesforce objects) via a System API to dynamically fetch field structures. In this platform’s design, however, the connectors are “Spine-aligned” and come with a known mapping to Spine entity types, so the initial schema is likely populated from internal knowledge rather than live API calls[42] [43]. Nonetheless, the connectors selected do influence the schema: for example, if the user’s chosen connectors include systems that manage certain data (say Tickets from a support system), the initializeSpine Process API can ensure that the Ticket entity type (and its core fields) are added to the tenant’s schema registry so the pipeline knows to expect and allow them.
•	Schema Router Resolution: After seeding, the platform’s Schema Router logic computes the resolved schema for the tenant by merging the baseline schema with any tenant-specific extensions[44] [45]. Initially, right after onboarding and before any data sync, the tenant-specific extensions might just be those seeded from the baseline and known connector structures. Over time, this will include new fields or even new entity types observed from actual data (more on this in step 9 and 10). The resolved schema determines the allowed entity types and fields for the tenant’s data pipeline. This resolved canonical model is effectively the contract for what data will be stored in the Spine (SSOT) for this tenant.
•	Schema Ready Before First Writes: While the initial sync can be triggered immediately after OAuth (Step 4), the key requirement is that schema resolution is available before records reach the Filter/Validator stages and before any records are written to Spine. In practice, initializeSpine and Schema Router resolution complete before (or at least in parallel with) the first sync writes, so the pipeline always has an authoritative allowed-types/fields contract when processing incoming records.
Logical completeness: The sequence correctly includes adaptive schema generation. A key detail is clarifying when and how the canonical schema is hydrated: the canonical schema (Spine model) is first established during onboarding using the Schema Picker inputs, and then formally initialized/seeded via initializeSpine before data loading. This ensures that the pipeline has the necessary schema context (allowed entity types, expected field structures) as soon as data starts flowing. There is no gap here, but it’s worth emphasizing that schema resolution precedes data classification and normalization. The Spine Schema Registry is in place from the start of data ingestion, and the Schema Router uses it together with baseline schema to guide the next steps[46] [47].

6. Type Classification and Schema Enforcement
With the schema resolved, the system now knows which entity types (data objects) and fields are relevant for the tenant. As the initial data sync proceeds, the integration must determine each incoming record’s type and ensure it fits the allowed schema:
•	Data Type Classification: When the Loader retrieves data from an external system, either the connector itself tags each record with an explicit entity_type or the integration pipeline performs a classification step to identify what kind of entity the data represents. For example, if a Salesforce connector returns a JSON object, the pipeline (in its “Classify” stage) might examine the payload to determine if it is a Contact, Account, Opportunity, etc., unless the connector has already indicated the type. This Type Classifier logically operates within the pipeline’s early stages[48]. It uses the context of the connector and possibly the content of the data to assign a canonical entity type to each record.
•	Enforcing Allowed Entity Types: Immediately after classification, the pipeline’s Filter stage enforces the tenant’s schema constraints. Using the resolved schema from the Schema Router (which includes baseline plus any tenant-specific extensions), the Normalizer checks if the classified entity_type of each record is in the tenant’s allowed list[49] [50]. If a record type is not permitted for this tenant (for instance, a connector returns an entity that falls outside the chosen domain/industry scope), the integration will drop the record and not process it further[51]. This prevents “rogue” or irrelevant data from polluting the Single Source of Truth. Essentially, the tenant’s Spine configuration (SSOT) acts as a gatekeeper in real time.
•	Timing of Schema Resolution vs Classification: To answer the specific question – the schema resolution (via Schema Router) occurs before and during data loading. The baseline schema is established at onboarding, and the Schema Router is consulted at runtime to get the latest allowed types (which initially equals the baseline set). The Type Classification of each record happens within the pipeline (Loader/Normalizer stage), after the schema has been resolved. In effect, the pipeline classifies a record’s type, then immediately uses the pre-computed allowed-type list from the Schema Router to decide if that record’s type is acceptable[52]. If not, it’s filtered out; if yes, processing continues. This ensures that classification informs what the record is, and the schema (already hydrated with baseline + dynamic definitions) informs whether to keep and store it.
•	Connector Metadata Influence: The metadata of connectors (i.e., the data categories they provide) is indirectly used in this phase by informing the schema. Since connectors were offered based on the selected industry/department, the data types they produce should generally align with the allowed types. If a connector can produce multiple entity types, the platform’s Schema Router has included all those relevant types in the allowed set if they correspond to the selected domain. If a connector tries to send data outside the expected types (which is unlikely if the catalog filtering was accurate), the filter will drop it. In a MuleSoft integration, this is akin to using System API metadata and Process API logic to enforce that only certain data objects flow through for a given customer or context, a practice which prevents, say, an HR system’s data from ending up in a Sales domain’s data store.
Logical completeness:
•	Ordering is sound: allowed types/fields are resolved first (onboarding + registry), then records are classified and immediately filtered against that schema.
•	Where it happens: “Type classifier” is not a separate system; it’s an internal early stage of the Loader/Normalizer pipeline.
•	Key control: tenant-specific schema rules are enforced in-pipeline via Filter/Validator so irrelevant entity types never reach Spine.

7. Loader: Retrieving Data (8-Stage Pipeline Execution)
With authorization in place and schema configured, the integration now loads data from the external systems. The Loader component is responsible for initiating and managing the data retrieval and processing pipeline. According to the design:
•	Bounded Data Fetch (Phase “Creamy”): The Loader (which can be seen as part of the Process API layer for data integration) invokes the connector’s data extraction, typically specifying parameters like phase (creamy for the initial lightweight sync), date ranges or cursors, and limits to enforce a bounded fetch[54] [55]. This aligns with best practices to avoid large initial data dumps. For example, the first sync might request only the last 7 days of data or the top N records (e.g., top 100 opportunities) to quickly provide value to the user.
•	Running the 8-Stage Pipeline: The Loader receives the raw data payload from the connector and then executes the eight-stage processing pipeline on this data[56]. These stages include:
a.	Analyze – Inspect the raw payload structure and content.
b.	Classify – Determine the entity type of each record (as discussed in Step 6 above).
c.	Filter – Apply tenant-specific filters (drop records of types not in the allowed schema, as determined by Schema Router)[57].
d.	Refiner – Clean or transform the data (e.g., normalize formats, handle missing values).
e.	Extractor – Extract sub-components or nested data if needed (e.g., flattening structures or pulling out embedded entities).
f.	Validator – Validate the data against schema constraints and business rules.
g.	Sanity Scan – Perform any final checks (e.g., PII scrubbing or data quality enforcement).
h.	Sectorizer – Prepare the data for storage by organizing it into the correct domain sectors (this often means partitioning or tagging data by domain/entity type for writing to the right place in the Spine). The output of this stage is typically a set of normalized entities ready to be written into the unified store.
•	Pipeline Orchestration: The 8 stages can run in-memory or via a queue. In this design, the Loader either runs the pipeline in-process or enqueues the data to a PIPELINE_QUEUE for asynchronous processing by the Normalizer service[58]. In either case, the Loader is ensuring the pipeline logic executes fully. Importantly, the Loader is aware of the tenant’s configuration – it passes the tenant’s allowed domain keys (from tenant_spine_config) into the pipeline context at the very start[59]. This means from stage 1 onward, the pipeline knows which entity types and domains are permitted for this tenant, enabling early decisions on data relevance.
•	No Parallel Datastores: All through this process, data is kept in-memory or in-transit; the Loader/Normalizer do not persist the raw external data to any intermediate database. This prevents the creation of shadow databases. The only target for structured data is the Spine (the SSOT). In MuleSoft terms, the Loader and Normalizer act as part of an System API/Process API hybrid – they transform and deliver data to the System API (Spine service) for storage.
Logical completeness:
•	Complete: Loader performs bounded fetch (Phase 1 “creamy”) and runs (or queues) the 8-stage pipeline with tenant context injected up front.
•	Clarification: Filter/Validator are the enforcement points for schema constraints derived from Schema Router + registry.
•	Note: It’s typical to follow “creamy” with “needed” and then continuous “delta” syncs to balance time-to-value vs completeness[60].

8. Normalizer: Tenant-Specific Canonical Mapping and SSOT Write
As the pipeline processes data, the Normalizer component ensures that outgoing data from the pipeline is properly formatted and inserted into the Spine (the Single Source of Truth):
•	Tenant Filtering: Building on the Filter stage mentioned earlier, the Normalizer enforces multi-tenancy and schema compliance rigorously. Each record carries a tenant_id and an entity_type as it moves through the pipeline. The Normalizer confirms the record’s tenant_id matches the current tenant’s context and that the entity_type is allowed for this tenant. This uses the allowed types list from the Schema Router (which was derived from the tenant’s tenant_spine_config and updated schema registry)[61]. Any data not conforming is discarded before storage.
•	Canonical Field Mapping: The Normalizer transforms each record from the source-specific format to the canonical data model format used by the Spine. In practice, this involves using mapping rules or DataWeave transformations (in a MuleSoft setting) to rename fields and restructure the object into the Spine schema shape[62]. For example, a Salesforce Account.Name might be mapped to the canonical field account_name in the Spine’s account entity, and a HubSpot DealAmount might map to a standard opportunity_value field for deals.
•	Sectorize and Prepare for Write: The Normalizer finalizes each cleaned and mapped record into a normalized Spine Entity format – typically an object with fields like entity_type (e.g., “Account”), a primary name or identifier, a source and source_id to trace back to the origin system, and the rest of the record data under a standard data structure[63]. The Normalizer then invokes the Spine System API to write this entity to the database.
•	Writing to the Spine (SSOT): The write operation (e.g., an internal API call or direct database operation) stores the normalized data into the tenant-specific domain tables in the Spine (for example, an Account record goes into the spine.account table, tagged with the tenant_id)[64]. The Spine, implemented on a scalable SQL database (here, Supabase/Postgres), is the central data store where all integrated data resides for querying and further processing.
•	Updating the Schema Registry: A vital behind-the-scenes action occurs during the write: as each record is written, the system checks for any new fields or entity types that were not previously seen in the schema. If any are detected, the Spine service logs an observation in the spine_schema_registry for this tenant[65]. This is the mechanism by which the Adaptive Schema evolves:
o	If a new field appears in the data (say, a custom field in Salesforce that wasn’t part of the baseline schema), the act of writing it to the Spine triggers a record of that field in the registry for this entity type[66]. The next time the Schema Router runs (which can be on-the-fly for each request or at least prior to the next sync cycle), it will include that field as part of the tenant’s schema. This means the canonical model “learns” from incoming data.
o	If an entirely new entity type appears (which is less common if connectors are chosen based on allowed domains, but possible if the baseline schema didn’t include something), the system can similarly record it. In practice, during initial sync the allowed entity types are those from the baseline schema (since new types would be filtered out on first pass). However, if for some reason a connector was expected to bring a new type, the platform could incorporate it by updating the allowed list before second-phase syncs. In the current architecture, new entity types are effectively added to the allowed set once observed and recorded in the registry, influencing subsequent sync phases[67] [68]. This ensures no data is permanently lost; it might require a second sync to capture what was initially skipped, unless the connector’s Phase 2 (“Needed”) accounts for it.
•	Single Source of Truth: After this step, the tenant’s Spine database now contains the normalized data from the external systems, all under one roof (with each record stamped with the tenant ID and source info). This completes the initial integration of structured data (Flow A) for the new workspace[69]. The Spine now holds the key business entities aggregated from the connected apps, ready to be used by the application’s features.
Logical completeness:
•	Complete: Normalizer enforces tenant_id + allowed entity types, maps to the canonical Spine model, and writes to Spine.
•	Clarification (important): The Spine Schema Registry (spine_schema_registry) is updated during writes by logging observed fields/types—this is the mechanism enabling adaptive schema[70] [71].
•	Implication: Schema Router can incorporate new fields on subsequent resolutions without requiring upfront schema redesign.

9. Populating the Dashboard from the Spine
With the Spine now containing the initial set of integrated data, the platform can populate the user’s workplace dashboard. This occurs once onboarding is completed (including the connector setup and initial sync). Key points:
•	Workspace Initialization Complete: The user interface transitions from the onboarding wizard to the main application (often called the WorkspaceShell in this platform)[72]. The navigation and modules available are configured based on the now-populated tenant_spine_config (for example, showing a Sales domain with relevant menu options if the user chose Sales)[73] [74]. This is handled by the front-end reading the tenant configuration via the BFF (Experience API) tenant-config endpoint or potentially embedded in the final response of initializeWorkspace[75].
•	Retrieving Dashboard Data: The front-end calls the BFF (Experience API) to get data for dashboard widgets. The BFF (acting as the aggregation/orchestration layer) queries the Spine System API or database for the requested data. For example, a request for a sales pipeline summary might result in the BFF querying the spine.opportunity and spine.account tables (filtered by the tenant_id) and applying some summary logic.
•	Use of Cached/Local Data: Because the data has been normalized into the unified Spine store, the dashboard does not need to make expensive calls to external APIs in real-time for initial load. Instead, it uses the data that was fetched during the initial sync (and any subsequent syncs) that is stored in the platform’s database (this can be considered a cached copy of the external data, kept up-to-date via the integration)[76]. This design is aligned with best practices: reading from a local System of Record (the SSOT) is faster and more reliable than calling multiple external services live for each dashboard load. It also avoids hitting external API rate limits or incurring unnecessary load.
•	Data Composition: The BFF may also combine data from multiple internal sources. For instance, some dashboard elements might pull from the Spine (structured data) and others from the Knowledge store (unstructured documents or insights) or from the results of analytical computations (like a readiness score). All these would also be stored or computed within the platform’s boundaries (e.g., a “readiness” service might calculate KPIs from Spine data). The BFF aggregates these as needed, ensuring the data presented is consistent.
•	Spine as the Primary Source: The core records shown on the dashboard (customer lists, sales pipeline, recent support tickets, etc.) are served from the Spine’s domain tables and related tables (which are part of the Spine schema). For example, a “Recent Contacts” list module would query the spine.contact table for the tenant’s records. This was not a gap in the original sequence, but it is important to stress that the dashboard reads from the Spine (and related internal stores like the Knowledge base for documents) via the BFF[77] [78]. There is no direct reading from external systems at this point.
Logical completeness: The dashboard reads from the platform’s internal stores (primarily Spine) via the BFF (Experience API); “cached” here means Spine holds a local, normalized copy of data originally sourced from external systems. No direct external API reads are required for dashboard rendering.

10. Real-Time Sync and Event-Driven Updates
After the initial data load, the integration doesn’t stop. To keep the dashboard and other features up-to-date, the platform continues to synchronize with external systems in near real-time. This corresponds to incremental syncs and event-driven updates (often referred to as Phase 3 “Delta” syncs in a multi-phase approach[79]). Key aspects:
•	Incremental Sync Cycles: The platform likely schedules background jobs or uses event triggers to periodically call the connectors’ System APIs for updates. In MuleSoft terms, Process APIs might orchestrate scheduled retrievals, and System APIs interface with each external system to fetch data changes since the last sync. For instance, after the “creamy” initial sync, a “needed” sync might pull a larger dataset within a day, and then continuous delta syncs fetch new or updated records (e.g., new CRM records or updated tickets) at regular intervals[80]. These follow the same Loader → Pipeline → Normalizer → Spine process (Flow A) as the initial load, just with the phase set to “delta” and using saved timestamps or cursors to get incremental changes.
•	Event-Driven Updates: Where supported, the platform uses webhooks or streaming APIs for instant updates. For example, if a connector like Salesforce is configured for webhooks or Platform Events, a change in Salesforce (such as a new Contact added) can trigger a call to a designated endpoint in the integration (a webhook listener in the connector’s System API). This incoming event would directly invoke the Loader/Normalizer pipeline for that single record, leading to a near-instant write to the Spine without waiting for the next scheduled poll. This design wasn’t explicitly described in the original sequence, but it’s a common real-world addition for achieving real-time sync. If webhooks are not available, the platform falls back to polling at a reasonable frequency.
•	Schema Registry Updates on the Fly: As these ongoing syncs run, any new fields or objects discovered will continue to update the spine_schema_registry for the tenant in real time, using the same mechanism of logging field observations during the Spine write step[81]. This means the Adaptive Schema continues to evolve with each sync. The Schema Router will incorporate these changes, and the Normalizer will automatically allow new fields (and even new entity types, if added to the allowed list) on subsequent sync runs[82] [83]. In practice, if a connector introduces a new field after a software update or if an admin adds a custom field in the source system, the next sync will capture this field and add it to the registry. Administrators might need to map or surface the field in the UI, but the data will at least be stored.
•	Continuous User Experience: From the user’s perspective, after the initial setup, data on the dashboard will keep updating automatically. The front-end can use techniques like WebSocket updates or periodic refresh calls to the BFF (Experience API) dashboard endpoint to show new or changed records. Because the integration is handling data sync in the background, these updates can often be delivered with low latency. For example, if a new sales opportunity is created in the CRM, the user might see it appear on their dashboard moments later, fetched via the next delta sync or received via webhook and then read from the Spine.
•	Flow B and Flow C (Other Data Flows): In addition to the connector-based Flow A, the system likely also runs Flow B (unstructured data ingestion) and possibly uses Flow C (AI/Knowledge UI interactions), as indicated in the architecture[84]. While these were not the focus of the question’s sequence, for completeness:
o	Flow B (documents & emails, etc.): If the user also ingests files or emails, a similar Loader/pipeline process applies. Files might be uploaded or fetched via connectors, then an extraction pipeline runs and relevant information is stored in the Spine (e.g., text extracted into context_extractions table) and in a separate Knowledge store for full-text search and AI use[85] [86]. Flow B ensures unstructured data is linked to the same entities in the Spine (through entity linking) so it can be shown on the dashboard and used in analyses.
o	Flow C (AI chat sessions/Knowledge UI): If the user interacts with AI features (like an AI assistant that generates summaries or insights), those are not written to the Spine. Instead, they go to a transient store (D1) and a specialized Knowledge UI store (for user-curated content) as described in the design. These are kept separate from the Spine, unless the user explicitly allowed linking them (via the linkKnowledgeUIToEntity360 flag) in which case the AI’s outputs can be considered in analytical views (read-only) but still not written into the Spine[87] [88]. This separation prevents unverified AI-generated data from contaminating the system of record (Spine).
Logical completeness: The continuous sync phase is complete; the key implementation clarifications are:
•	It’s important to note how external updates reach the system: either by scheduled pulls (delta syncs) or by push events (webhooks), depending on connector capabilities.
•	Additionally, a potential implied step is the management of sync state: the integration must maintain checkpoints (e.g., last synced timestamp or an incremental token for each connector) to know what data to pull in delta cycles[89] [90]. This detail is generally handled by the Process/System API layer (e.g., storing since timestamps in a database or cache).
•	Finally, ensure that the Spine’s data remains the source for the dashboard during continuous updates. The front-end should not directly call external APIs when, say, the user hits “refresh” on a dashboard widget; instead it triggers an update in the integration layer which then updates Spine, and the refreshed data is fetched from Spine.

Conclusion and Verification of Flow Completeness
The above analysis confirms that the integration data flow (steps 1–12) is largely consistent with a robust, real-world API-led integration pattern. The flow, in the context of MuleSoft’s layered APIs, involves the BFF (Experience API) (UI-facing endpoints for auth, onboarding, connector setup, and dashboard), Process APIs (orchestration of onboarding, data synchronization logic, schema management), and System APIs (connectors to external systems like Salesforce/HubSpot, and internal system access like the Spine database) working in concert.
Identified Gaps/Clarifications: The main elements that needed clarification were:
•	Canonical Schema Hydration: Occurs during onboarding via the Schema Picker and initializeSpine call, which seeds the Spine schema for the tenant before data flows[91] [92].
•	Connector Metadata & Schema Adaptation: Connector capabilities (industries/departments and data objects) are used to filter connector selection and seed the schema. New fields from connectors are captured at runtime and added to the schema registry for adaptation[93] [94].
•	Type Classifier Timing: The data type classification (pipeline “Classify” stage) is part of the data loading pipeline (Loader/Normalizer) and is executed after the baseline schema is in place; it uses the previously resolved schema to inform classification and subsequently filtering of entities[95].

•	Spine Schema Registry Updates: Yes, the registry is updated during sync writes to Spine, ensuring the schema evolves as new data is integrated (no gap here; this is explicitly part of the design)[97].
•	Dashboard Data Source: The dashboard exclusively uses the internal Spine (and related stores) via the BFF (Experience API) endpoints for data, not direct external API calls, which is the intended design for performance and data consistency[98] [99]. This is logically sound.
Having addressed these points, the integration flow is end-to-end complete: from user landing and authentication, through onboarding and schema setup, connector integration and data sync (Flow A), to feeding the unified Spine and displaying the dashboard, with ongoing syncs to keep data fresh.
